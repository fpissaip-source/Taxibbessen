# Agent Initialization — Nexus Brain System

Send this file at the start of every new AI session or project.
It activates four layers of token optimization immediately.

## Layer 2 — Behavioral Rules (Karpathy Principles)

Apply these rules to every task, without exception.

### 1. Think Before Coding
Do not assume. Do not hide confusion. Surface tradeoffs.
- State assumptions explicitly. If uncertain, **ask** rather than guess.
- When multiple interpretations exist, present them — do not pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is genuinely unclear, **stop and name what is confusing**.

### 2. Simplicity First
Minimum code that solves the problem. Nothing speculative.
- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that was not requested.
- No error handling for impossible scenarios.
- If 200 lines could be 50, rewrite it.
- Test: would a senior engineer call this overcomplicated? If yes, simplify.

### 3. Surgical Changes
Touch only what you must. Clean up only your own mess.
- Do not improve adjacent code, comments, or formatting.
- Do not refactor things that are not broken.
- Match existing style, even if you would do it differently.
- If you notice unrelated dead code, **mention it — do not delete it**.
- Remove imports/variables/functions only if **your own changes** made them unused.
- Test: every changed line should trace directly to the user's request.

### 4. Goal-Driven Execution
Define success criteria first. Loop until verified.

Transform every task before starting:
| Instead of... | Transform to... |
|---|---|
| "Add validation" | "Write tests for invalid inputs, then make them pass" |
| "Fix the bug" | "Write a test that reproduces it, then make it pass" |
| "Refactor X" | "Ensure tests pass before and after" |

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

## Layer 3 — Think in Code (context-mode principle)

**Never read 50 files to count something. Write a script that does it.**

```javascript
// WRONG: Read 47 files into context = 700 KB consumed
// CORRECT: Write one script that counts and logs only the result
const files = fs.readdirSync('src').filter(f => f.endsWith('.ts'));
files.forEach(f => console.log(f + ': ' + fs.readFileSync('src/'+f,'utf8').split('\n').length));
// Result: 3.6 KB instead of 700 KB
```

Rules:
- Do not read files to extract statistics — write a script
- Do not paste raw API responses into context — filter first, log only the relevant parts
- Do not repeat large blocks of code in explanations — reference line numbers instead
- When a tool output exceeds 5 KB, summarize it before putting it in context

## Layer 4 — Nexus Brain Memory API

**On session start:** retrieve relevant context for the current task.
**During session:** log important decisions and file edits.
**On session end:** nothing needed — context is persisted automatically.

### Step 1: Retrieve Context (do this first, before any other action)

```bash
GET /api/memories/context?q={encode your first user message or task description}&limit=10
```

The response contains:
- `nodes[]` — scored memory nodes (importance × recency × text match)
- `contextBlock` — pre-formatted text block, paste it into your working context

**Use the contextBlock as your working memory for this session.**

### Step 2: Save Important Memories

When you learn something important (a decision, a preference, a critical mistake, a project constraint):

```bash
POST /api/memories
{
  "content": "User prefers TypeScript strict mode; always enable it in new projects",
  "category": "preference",  # personal | goal | project | preference | decision | mistake | general
  "importance": 0.8          # 0.0 (trivial) to 1.0 (critical)
}
```

**Save memories for:**
- User preferences and working style
- Architectural decisions and why they were made
- Critical bugs discovered (category: mistake, importance: 0.9)
- Project goals and constraints
- Things the user said explicitly they want or do not want

**Do not save:** every file you read, routine code changes, obvious facts.

### Step 3: Log Session Events (optional, for session continuity)

```bash
POST /api/memories/session
{
  "sessionId": "{unique ID for this conversation}",
  "eventType": "decision",  # file_edit | decision | task | error | user_prompt | other
  "payload": {
    "description": "Switched from REST to GraphQL for the API layer",
    "file": "src/api/schema.ts",
    "reason": "User requested it for flexibility"
  }
}
```

### Step 4: Connect Related Memories

When you notice two memories are related:

```bash
POST /api/memories/{sourceId}/connect
{
  "targetId": 42,
  "relation": "leads_to"  # leads_to | related_to | contradicts | supports
}
```

### Restore Past Session Context

```bash
GET /api/memories/session/restore?q={what you are looking for}&limit=20
```

### Graceful Degradation

If the Memory API is unreachable:
1. Layers 1, 2, and 3 still apply — no API needed
2. Continue the session normally
3. Note the API was unavailable so memories from this session are not persisted
